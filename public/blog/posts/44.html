Advent of Code 2025: C++

Day 06: Trash Compactor

2025-12-06

It's the weekend, so today's probably gonna be a step up in difficulty! Indeed, this is another puzzle where basically the whole point is to parse the input, but it's even harder than before...

Part 1

I've been avoiding having to split a string by a delimiter for a while, but I think it's finally time to write one. my splitBySpaces() function takes a string str and runs std::getline on it to extract the initial substring until a ' ' space character. Since in the input the numbers are delimited by possibly multiple spaces, and there may be initial or final spaces, I discard this initial substring if it's empty. Thus I get myself a std::vector<std::string> object for each line. From there I can run the calculations fine.

Something new to me today was thinking about C++ as a compiled language, and trusting the compiler to optimize some stuff for you. In my solution to Part 1, in my logical thinking, I wanted to place the case-match statement (determining if we're adding or multiplying) outside the for loops which run the computations. My girlfriend Trice told me that that didn't matter, for some computer science/compilers reason I don't have the background for, so indeed my case-match statement is inside the for loop and it still runs very quickly.

Part 2

Oh, this part is evil! The first thing I did was to add a parameter to my ProblemSet class which would hold the raw strings. Next I'm finding the locations along the string of all the operations on the bottom row and storing them in. From this we can figure out the columns in which numbers are stored: My variable part2Problems is a std::vector<std::pair<int, int>>, representing the first and last columns of the problem in question. The symbol for the problem can be found pretty easily from the first column of the problem, and so it's essentially just up to me to implement the algorithms for reading the numbers from the columns.

Somehow this part went pretty much without a hitch! I'm glad because it's quite complex, but I reckon my habitual modularization has been paying off.
